负载均衡功能看似很简单----把请求转发给后端。
但在生产场景上，面对众多负载均衡算法，上游集群有10k个后端节点，考虑以下因素， 负载的代理精度(依据: 请求数， 活跃连接数，响应延迟), 后端差异(机器规格不同考虑加权算法), 确定状态(处于缓存、会话状态等固定打到同一机器), 可见范围(全局状态不可见的羊群效应和解决方案)

## 一、负载均衡的起源

负载均衡是被"单机瓶颈"逼出来的概念。早期单台机处理请求，硬盘上放着静态文件,来请求就发。"架构"一词还不存在。

瓶颈: 
1. 流量涨到CPU，网卡，连接数扛不住; 
2. 单机宕机了,整个站点没了...

方案:
1. 纵向扩展: "换台更强的",加CPU核、加内存。不改架构，不引入任何复杂问题。硬伤是成本非线性增长，投入不成比例，单机有上限，并且宕机问题还是没有解决。
2. 横向扩展: "加N台服务器",容量约等于N倍，单机宕机另一台顶上。新问题:怎么选机器？

把请求派发到横向机器池的这个决策，就是`负载均衡`。
这是横向扩展的入场劵，只要有一个池子，就必须有个代理解决分发问题。
负载均衡的岁月史书， 就是这个"派发者"一步一步变聪明且更灵活的历史。

> "单机模型"并非一定是落后的,它处理问题的简单方式至今仍然在互联网架构上发光发热。

## 二、DNS轮询
早期负载均衡器: DNS
(RFC 1794, 1995年)DNS已支持一个域名对应多个IP，即一个域名配多条A记录，解析时轮回顺序返回。
优雅，现成的负载均衡器。零新增复杂性。

缺点:
1. 故障延迟:DNS带有缓存，TTL没过期之前不会走解析路径，即不会负载均衡。 对机器故障的延迟反应慢！
2. 无状态负载:本质就是轮询IP地址，根本不知道真实的后端状态，机器挂了仍然转发，不会"下线"
3. 不支持复杂:轮询算法过于简单，受限于第2条，无法支持更多灵活的负载算法。

DNS负载均衡的缺点，反应慢，无状态，只支持轮询。 这就是负载均衡器应该作为独立的设备的诞生起点。

## 三、 负载均衡算法
负载均衡的"均衡"一词没有"唯一"的定义。 没有最优的负载均衡算法，这往往需要结合场景说话。

## 四、轮询，加权轮询和平滑加权

### 4.1 轮询(Round Robin)
轮询一句话: 维护一个游标，每来一个请求`i = (i + 1) % N`, 循环。
代理信号: 请求数
假设: 每个请求开销相等，每台后端机器相等。

```
// A: [1, n]
Round_Robin(A, n)
  idx <- A->counter % n
  A->counter <- A->counter + 1
  if A->counter > n:
    A->counter <- 0
  return idx
```

优点: 几乎免费, 一个整数计数器，$O(1)$,无状态，完全确定。
在请求开销同质，机器同质的场景，它比任何更聪明的算法效果一样。

当假设成立，引入其他负载算法是对复杂度的浪费，这是它的理念。
