## 第二部分：具体实操建议

### 1. 现在先不要做的

* 不要一开始上协程
* 不要一开始做 HTTP/2、RPC、复杂业务系统
* 不要把扩展项全塞进主线

### 2. 先做的最小闭环

按这个顺序落地：

1. 工程骨架
2. socket + epoll + EventLoop
3. ThreadPool + Scheduler
4. echo server
5. 最小 HTTP/1.1
6. Logger + Timer + Config
7. `/metrics` + Trace ID
8. wrk 压测
9. 扩展优化

### 3. 你的主线必须钉死

主线只保留：

* epoll 事件驱动
* Scheduler + ThreadPool
* Timer
* Async Logger
* Monitor
* HTTP/1.1 + Keep-Alive
* `/metrics`
* Trace ID
* 压测与图表

### 4. 扩展项只选这几个里最值得的

优先级建议：

1. **协程**
2. **监控页面可视化**
3. **请求全生命周期追踪增强**
4. **WebSocket demo**
5. **时间轮**

### 5. “运行时系统”这个题的真正亮点怎么做实

不要只写“借鉴操作系统思想”，要在代码里落地成：

* 任务状态
* 调度入口统一
* 超时与取消
* 任务队列与线程池协作
* 监控与追踪

### 6. 示例服务怎么定

主 demo：

* HTTP/1.1
* 静态文件
* JSON API
* Keep-Alive
* Trace ID 返回头

扩展 demo：

* WebSocket 回显或聊天室

### 7. 压测怎么准备

至少做这几组：

* 不同并发下的 QPS / P50 / P90 / P99
* Keep-Alive 开关对比
* 不同线程数对比
* 优化前后对比
* 扩展项启用前后对比

### 8. 文档写法上最重要的点

论文里不要说“我做了一个服务器”，要说：

* 设计了一个服务运行时系统
* 统一组织了事件驱动、任务调度、资源管理和可观测性
* 通过 HTTP 服务验证其实用性
* 通过压测验证其性能与稳定性

### 9. 你现在最该立即产出的东西

先立刻写出来：

* 项目目录结构
* 模块依赖关系图
* 阶段二最小闭环任务单
